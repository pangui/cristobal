Encontrado. Mira lo que veo en mi entorno ahora mismo:

```
CRISTOBAL_AUTOSTARTED=1

```

`CRISTOBAL_AUTOSTARTED=1` **está exportada en el entorno del Cursor server**. Eso significa que todo terminal integrado nuevo la hereda, y el tercer guard del bloque (`[[ -z "$CRISTOBAL_AUTOSTARTED" ]]`) bloquea el autostart **siempre**.

Causa probable: en algún momento lanzaste `arquitecto` (u otro alias) desde un shell que luego lanzó el Cursor server — la var exportada viaja hacia arriba por herencia y queda pegada en el server hasta reiniciarlo.

El guard `CRISTOBAL_AUTOSTARTED` además es redundante: `claude` ya exporta `CLAUDECODE=1`, que protege contra recursión mientras claude corre. La única razón de existir era cubrir `source ~/.zshrc` dentro de claude — caso que el guard de `CLAUDECODE` ya cubre.

Recomendación (la ordeno, una opción):

1. **Quitar el guard `CRISTOBAL_AUTOSTARTED` del bloque** (y el `export`), dejando solo `CLAUDECODE`. Lo edito en `main`, propago al runtime, y después reinicias la ventana de Cursor para limpiar la var fantasma del server.

Alternativa (lo que sacrifica): dejar el guard pero cambiar `export` por asignación local — no limpia el estado actual del Cursor server, así que requeriría reinicio igual **y** deja un guard cuyo único caso de uso ya está cubierto por CLAUDECODE.

¿Procedo con la primera? El cambio es:

```zsh
# antes
if [[ -o interactive ]] && [[ -z "$CLAUDECODE" ]] && [[ -z "$CRISTOBAL_AUTOSTARTED" ]]; then
  if [[ "$PWD" == "$HOME/cristobal/especialistas/"*  ]] && [[ -f "$PWD/ROLE.md" ]]; then
    export CRISTOBAL_AUTOSTARTED=1
    claude
  fi
fi

# después
if [[ -o interactive ]] && [[ -z "$CLAUDECODE" ]]; then
  if [[ "$PWD" == "$HOME/cristobal/especialistas/"* ]] && [[ -f "$PWD/ROLE.md" ]]; then
    claude
  fi
fi
```
